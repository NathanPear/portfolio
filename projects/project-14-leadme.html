PROJECT 14 / SOFTWARE

Leadme

A local-first, queue-driven lead discovery and outreach drafting service built with SQLite, Docker, and schema-validated JSON contracts.

  • Role: backend service design and workflow orchestration
  • Focus: serial queue workers and structured data validation
  • Constraint: local-first execution and strict prompt contracts

Project Details

Problem and Context

Nathan needed an automated, deterministic way to discover company leads, normalize profiles, enrich contact metadata, and draft outreach emails tailored to local mechatronics and automation roles near Hamilton, Ontario.

Rather than relying on manual copy-pasting or complex cloud subscriptions, the goal was to build a local operational service that runs reliably on a workstation or NAS-backed Docker stack.

Scope and Constraints
  • Local-first hosting using Docker Compose, SQLite, and local YAML configuration
  • Strict serial queue worker enqueuing exactly one job at a time to isolate state transitions
  • JSON-only prompt contracts with strict schema validation; no hidden agentic tool loops inside Leadme
  • Draft-only outreach (no automated sending in version 1 to keep a human-in-the-loop validation barrier)

Approach

What I Built
  • Six-Step Serial Queue: Designed a structured pipeline: Fisher (discovery), Librarian (normalization), Researcher (enrichment), Examiner (fit check), Writer (outreach drafting), and Critic (review).
  • Hermes OpenAI-Compatible Gateway: Standardized client interfaces speaking to local model runtimes, validating every JSON response against strict JSON schemas before committing mutations.
  • Relational SQLite Store: Engineered database structures tracking lead status, normalized company info, critic notes, and Gmail draft states.
  • Operator Dashboard: Lightweight HTTP API and operator dashboard to modify config settings, settings YAML, role prompt files, and inspect active queues.
Cross-Disciplinary Read

Leadme bridges software design patterns with systems engineering principles. The pipeline behaves similarly to mechatronic feedback loops: data inputs are verified against strict schema boundaries (sensor thresholds) before the system updates database state or enqueues downstream tasks.

By avoiding unpredictable agentic loops and constraining the runtime to a serial, single-worker database queue, Leadme shows how software workflows can be engineered with the same predictability and fail-safe boundaries expected of physical hardware.

Outcome

Result

The resulting service runs locally on Nathan's workstation, pulling companies, organizing database leads, and staging tailored email outreach drafts. Integration tests verify that enqueued jobs advance smoothly and tracking states update correctly.

The system passes its full smoke test suite (including test_health_and_fisher_enqueue and test_pipeline_runs_against_fake_endpoint), demonstrating production-grade reliability on simple workstation hardware.

Takeaways
  • Serial queues keep multi-step workflow pipelines predictable and simple to debug
  • Strict schema boundaries at every step prevent prompt leakage and bad model data from corrupting the store
  • Separating prompt files and context markdown from code files makes configuration simple for operators
