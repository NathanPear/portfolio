PROJECT 12 / SOFTWARE / INTEGRATED

Agentic Automation Infrastructure

A self-hosted, multi-agent environment orchestrating model-routing, containerized services, and structured automation workflows.

  • Role: independent systems integration and DevOps
  • Focus: agent infrastructure and container orchestration
  • Constraint: resource-constrained hardware and strict privacy limits

Project Details

Problem and Context

Moving from basic AI chat interactions to dependable local-first automation requires solving deployment, networking, token management, and reliable model fallbacks.

Running multi-agent systems on resource-constrained private hardware demands robust orchestration around models to ensure service reliability, data sovereignty, and secure tool access.

Scope and Constraints
  • Deploy a multi-service agent stack securely on a local host
  • Implement model routers handling fallbacks, caps, and timeouts
  • Integrate tools via profile-aware gateways separating read/write bounds
  • Support workflow automation with deterministic validation and state boundaries

Approach

What I Built
  • Docker Stacks: Separated containers by agent roles (Anna, Nightingale, Router) using Compose volume mappings and network isolation.
  • Local Policy Router: Smart model ladders, fallback policies, context cap enforcement, and YAML-defined configuration.
  • lazy-mcp Gateway: Dispatch routing and credential boundaries restricting tool exposure based on active profile scopes.
  • A2A System: A file-backed, deterministic agent-to-agent (A2A) message relay system with deduplication keys, TTL, and status transitions.
Cross-Disciplinary Read

This project integrates systems engineering, devops, and artificial intelligence into a coherent, self-hosted operational layer. A reviewer from a robotics or mechatronics background can easily see the parallel: the agent represents the controller, the MCP acts as the input/output bus, and containerized services are the physical actuators.

It also demonstrates strong systems-integration discipline: managing network boundaries, handling auth lifecycles, and treating LLMs as raw compute modules wrapped in validation code.

Outcome

Result

The final system successfully routes requests dynamically between local and cloud models while exposing secure, profile-aware MCP toolsets to the agent harness.

Security is maintained by restricting outgoing operations to intermediate, human-in-the-loop validation states, eliminating the risk of unreviewed autonomous actions.

Takeaways
  • System reliability is built around the model, not inside it, via schemas
  • Local model hosting (llama.cpp) requires careful context and speed tuning
  • Restricting critical actions to human-in-the-loop staging states ensures safe autonomous runs
